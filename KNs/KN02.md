# KN02: Cloud Computing & Infrastructure as Code

## 1. Teil  Cloud Service Modelle

### 1. Begriffe zuordnen

Hosting von virtuellen Maschinen → **IaaS**

Hier bekommt man virtuelle Server von einem Cloud-Anbieter und verwaltet diese selber.

Nutzung von Google Docs → **SaaS**

Man benutzt eine fertige Software über den Browser und muss nichts installieren.

Containerverwaltung mit Kubernetes → **CaaS**

Hier werden Container automatisch verwaltet und skaliert.

### 2. Unterschied zwischen IaaS und PaaS

#### IaaS (Infrastructure as a Service)

Bei IaaS bekommt man virtuelle Maschinen, Speicher und Netzwerk vom Anbieter.  Man muss aber selber das Betriebssystem installieren, Updates machen und die Software konfigurieren. Man hat viel Kontrolle, aber auch viel Verantwortung.

Beispiel: Amazon EC2

#### PaaS (Platform as a Service)

Bei PaaS kümmert sich der Anbieter um Server, Betriebssystem und Laufzeitumgebung.  
Man lädt nur seinen Code hoch. Das ist einfacher, weil man sich nicht um die Infrastruktur kümmern muss.

Beispiel: Azure App Service

#### Hauptunterschied

Bei IaaS verwaltet man die Server selbst.  
Bei PaaS verwaltet man nur die Anwendung.

## 2. Teil – Infrastructure as Code (IaC)

### Warum automatische Konfiguration besser ist

Automatisierung ist besser als manuelle Konfiguration, weil:

- weniger Fehler passieren
- alles dokumentiert ist
- Systeme schneller erstellt werden können
- man Umgebungen immer wieder gleich aufbauen kann

Wenn man alles manuell macht, vergisst man schnell etwas oder macht Tippfehler.

### Was bedeutet Infrastructure as Code?

Infrastructure as Code bedeutet, dass man Infrastruktur mit Code beschreibt.

Zum Beispiel werden Server, Netzwerke und Datenbanken in einer Datei definiert.  
Mit einem Befehl wird dann alles automatisch erstellt.

Das ist wie ein Bauplan für die IT-Infrastruktur.

### Tools für IaC

#### Terraform

Terraform wird benutzt, um Cloud-Ressourcen automatisch zu erstellen.

Man schreibt eine Konfigurationsdatei und Terraform erstellt z.B. Server oder Netzwerke bei AWS, Azure oder GCP.

Vorteil: Man kann alles versionieren und wiederverwenden.

#### Docker

Docker wird benutzt, um Anwendungen in Container zu verpacken.

Ein Container enthält die Anwendung und alle Abhängigkeiten.  
Dadurch läuft sie überall gleich, egal ob lokal oder in der Cloud.

### Kubernetes und Container-Orchestrierung

Kubernetes ist eine Plattform, um viele Container automatisch zu verwalten.

Container-Orchestrierung bedeutet:

- Container starten und stoppen
- automatische Skalierung bei hoher Last
- Neustart bei Fehlern
- Updates ohne Ausfall

Kubernetes wird vor allem in grösseren Umgebungen eingesetzt.

## 3. Teil – Globale Cloud-Plattformen

### Vergleich der Infrastruktur

| Plattform | Regionen | Verfügbarkeitszonen |
|------------|------------|--------------------|
| AWS | 31 | 99 |
| Azure | 60+ | mehrere pro Region |
| GCP | 38 | 100+ |

Regionen sind geografische Standorte.  
Verfügbarkeitszonen sind getrennte Rechenzentren innerhalb einer Region.

### Vergleich von Compute-Diensten

| Anbieter                     | Compute-Dienst            | Beschreibung                         |
|-------------------------------|---------------------------|--------------------------------------|
| Amazon Web Services (AWS)    | Amazon EC2                | Virtuelle Server in der Cloud       |
| Microsoft Azure              | Azure Virtual Machines    | Virtuelle Maschinen in der Cloud    |
| Google Cloud Platform (GCP)  | Google Compute Engine     | Virtuelle Maschinen von Google      |
## Fazit

Cloud-Modelle unterscheiden sich darin, wie viel Verantwortung man selber übernimmt.

Infrastructure as Code hilft, Systeme automatisch und sauber aufzubauen.

Docker und Kubernetes sind wichtig für moderne Cloud-Umgebungen.

AWS, Azure und GCP bieten ähnliche Grundfunktionen, unterscheiden sich aber im Detail.
